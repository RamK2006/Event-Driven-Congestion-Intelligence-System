📋 Methodology & Limitations — Required Disclosure

⚠️ Honest Reframing — Critical Disclosure

This system does not predict traffic congestion, vehicle counts, or delay duration in minutes of traffic, because the dataset contains no traffic flow, speed, or volume measurements. Instead, it predicts three real, label-backed outcomes — event severity, road-closure likelihood, and clearance time — and uses historical spatio-temporal patterns to recommend resource deployment and diversion zones. This is a deliberate, disclosed scope decision, not a limitation discovered after the fact.

🎯 What This System Predicts

  • Priority Classification (High/Low) — Predicted the moment an event is reported, using LightGBM classifier on spatial, temporal, and operational features.
  • Road Closure Requirement (Yes/No) — Predicted at report time using the same model family.
  • Clearance Time (minutes) — How long the disruption will last, predicted via LightGBM regressor on the subset of events with known outcomes (closed/resolved status only).

🗺️ Hotspot & Workload Intelligence

Geographic hotspots are identified using HDBSCAN density-based clustering on latitude/longitude coordinates. This finds genuine incident concentration zones independent of administrative boundaries. Police station workload is computed from rolling historical event counts.

🔀 Diversion Recommendations

Diversion suggestions are heuristic and co-occurrence-based, not network-routing-based. For each corridor, we identify which other corridors historically had concurrent events and recommend alternatives that historically did NOT have simultaneous incidents. Since route_path is 98.3% empty and no road-network graph data exists, this is explicitly a heuristic layer.

⚠️ Known Limitations

1. Clearance-time model is trained on only ~39% of the dataset (closed/resolved events). Active (ongoing) events are excluded as right-censored data — including them would bias the model toward underestimating duration.

2. Event coverage: 94.3% of records are unplanned, 5.7% planned. The system is validated primarily on unplanned incidents.

3. Zone/Junction coverage: Zone is 42% populated, junction is 31% populated. These are used as secondary features with explicit "known/unknown" flags.

4. Priority labels reflect operational/subjective police judgment, not an independent severity measurement.

5. Holiday calendar uses a static hardcoded reference table of Indian national and Karnataka state holidays — this is not an external dataset but a fixed calendar fact.

🏗️ System Architecture

  • Data Layer: Cleaned, feature-engineered dataset with documented exclusions
  • Modeling Layer: 3 LightGBM models + HDBSCAN clustering + heuristic diversion lookup
  • Inference API: Flask REST API accepting new incident reports
  • Dashboard: This Event Command Center with live incident map, prediction form, workload panel, trends, and methodology disclosure
  • Post-Event Learning: Rolling hotspot/workload statistics recalculated as new records close

📦 Dataset Summary

  • Source: Astram Event Data (Anonymized) — 8,173 records of traffic-disrupting events in Bengaluru
  • Features used: Event type, cause, location (lat/long, corridor, zone, junction, police station), temporal (hour, day, month, weekend, holiday), vehicle type, rolling event counts
  • Features dropped: 27 columns with >90% missing data, high-cardinality IDs, admin metadata
  • No external data: No APIs, scraped data, or internet-sourced datasets used (holiday calendar is a static reference table)
